# Game Creation Framework Design

**Date:** June 1, 2026 (design) | **Updated:** June 11, 2026
**Branch:** feature/game-management

> **Status update (June 2026):**
> - Registry unification is **complete**: `lib/game-pool/gameRegistry.ts` deleted, `lib/games/registry.ts` is single canonical registry. Section 2's "too much friction" list is shorter now.
> - `open_the_door` was **successfully added** as the worked example in Section 6. The contract/runtime/builderPreview structure proposed here is exactly what was built.

## Goal

Define an ideal lightweight framework for adding new games to the platform, where the only required deliverables are:

1. `contract.ts` — game metadata and behavior contract
2. `runtime.tsx` — actual play/runtime component
3. database seed — persisted admin configuration for the game

Everything else should be automatic or generated by convention.

## Current State

Today (post-registry-unification), adding a new game requires changes in:

- `lib/games/types.ts` to extend the `GameType` union
- `lib/games/registry.ts` to register the new contract
- `lib/games/<game>/contract.ts` for metadata and runtime wiring
- `lib/games/<game>/runtime.tsx` for the actual UI
- `lib/games/<game>/builderPreview.tsx` (recommended to avoid preview fallback to Mystery Box)
- `supabase/migrations/..._super_admin_games.sql` or a new migration to seed a `games` row
- `lib/builder/types.ts` to update `BuilderGameType`
- `components/promotion-builder/GameSelectionSection.tsx` — click handler has explicit allow-list
- ~~`lib/game-pool/gameRegistry.ts`~~ ← **DELETED** (registry unification complete)

The `lib/game-pool/gameRegistry.ts` requirement was removed. The other UI surface friction remains.

That is still too much friction for a platform designed to support many new game experiences.

## 1. If Open The Door were added today, which files must change?

In the current repository, the exact minimum files are:

- `lib/games/types.ts`
  - Add a new literal for `GameType`, for example `open_the_door`
- `lib/games/open-the-door/contract.ts`
  - Create the contract metadata and default values
- `lib/games/open-the-door/runtime.tsx`
  - Implement the game-specific play experience
- `lib/games/registry.ts`
  - Add the new contract to `gameRegistry`
- `supabase/migrations/20260430170000_super_admin_games.sql`
  - Add a seeded row for the new game

Depending on current builder and runtime plumbing, there may also be additional changes in:

- `lib/game-pool/types.ts`
- `lib/game-pool/gameRegistry.ts`
- builder UI components if they contain hard-coded game-type logic

## 2. Which changes are boilerplate?

The following changes are mostly boilerplate:

- `lib/games/types.ts`
- `lib/games/registry.ts`
- `supabase/migrations/..._super_admin_games.sql`
- any game-type union updates in legacy helper files such as `lib/game-pool/types.ts`

These files are structural scaffolding rather than game-specific behavior.

## 3. Which changes can be automated?

The following changes should be automated in the target architecture:

- registry registration
  - auto-import all `lib/games/*/contract.ts` modules
  - build `gameRegistry` from discovered contracts
- type generation
  - derive `GameType` union from the contract module keys / filenames
- builder selection UI
  - render from `availableGames` automatically
- play/runtime mapping
  - resolve `game_type` to contract and runtime component automatically
- database seeding
  - generate or validate `games` rows from contract metadata
- admin metadata sync
  - surface registry-provided name/icon/description in game management UI

## 4. What is the minimum set of files required for a new game?

In the ideal framework, the minimum files are:

- `lib/games/<game-name>/contract.ts`
- `lib/games/<game-name>/runtime.tsx`
- a Supabase migration or seed file to add a `games` record

Optionally, if the game needs custom builder preview UI:

- `lib/games/<game-name>/builderPreview.tsx`
- `lib/games/<game-name>/configPanel.tsx`

But these should be purely optional extras, not required for registration.

## 5. Target Architecture Proposal

### Core principles

- One canonical game registry.
- One canonical game type identifier.
- Game metadata and runtime behavior live together in a contract package.
- Database rows reference a stable `game_type`, not an arbitrary free-form slug.
- The admin UI, promotion builder, and play page all read from the same registry.
- New games should be discoverable by convention, not manual wiring.

### Ideal structure

1. `lib/games/<game-name>/contract.ts`
   - Exports a `GameContract` object.
   - Includes: `type`, `name`, `icon`, `availability`, `createCard`, `labels`, `analytics`, `PlayComponent`, and optional builder components.
   - Defines default `game_config` structure for the game.

2. `lib/games/<game-name>/runtime.tsx`
   - Implements the actual game UI/player component.
   - Is referenced by the contract (e.g. `PlayComponent`).

3. `lib/games/index.ts`
   - Auto-discovers all `lib/games/*/contract.ts` modules.
   - Exports `gameRegistry`, `availableGames`, and helper functions.
   - Optionally generates a `GameType` union from registry keys.

4. `supabase/migrations/..._seed_games.sql`
   - Adds the new game row to `games` with `game_type`.
   - Uses contract metadata to populate defaults.

### Auto-generated wiring

- `GameType` is derived from registry keys or filenames.
- `availableGames` and builder selectors render all non-hidden games automatically.
- `getGameDefinition(gameType)` resolves the contract from the registry automatically.
- `GameRuntimeRenderer` can use the same registry rather than a separate `lib/game-pool` map.
- Promotion creation and play-side code only need `promotion.game_type` and the registry lookup.

### Database design recommendation

Add or normalize a `game_type` column in the `games` table:

- `game_type` — stable identifier matching registry keys such as `open_the_door`
- `slug` — optional display slug, if needed for admin URLs or marketing

This removes the current mismatch between DB slugs and runtime game types.

## 6. Worked Example: Open The Door

### Desired new game package

Create a directory:

- `lib/games/open-the-door/`
  - `contract.ts`
  - `runtime.tsx`
  - optionally `builderPreview.tsx`
  - optionally `configPanel.tsx`

Because the framework is convention-based, no other files need manual edits.

### Example contract structure

`lib/games/open-the-door/contract.ts`

- `type: 'open_the_door'`
- `name: 'Open The Door'`
- `icon: '🚪'`
- `availability: 'active'`
- `createCard.title: 'Open The Door'`
- `createCard.description: 'Tap to open the right door and win an instant coupon.'`
- `labels.title: 'Open The Door'`
- `labels.instruction: 'Pick the right door to win your reward.'`
- `analytics.eventPrefix: 'open_the_door'`
- `resultDelayMs: 1200`
- `confetti: { particleCount: 180, spread: 80, origin: { y: 0.6 } }`
- `PlayComponent: OpenTheDoorRuntime`

### Example runtime component

`lib/games/open-the-door/runtime.tsx`

- Renders the door selection UI
- Accepts the standardized `GamePlayProps`
- Calls `onPlay()` when a door is selected
- Supports `rewards`, `canPlay`, `playing`, `playsRemaining`

### Example database seed record

`supabase/migrations/20260601120000_add_open_the_door_game.sql`

- Inserts a row into `games` with:
  - `game_type = 'open_the_door'`
  - `name = 'Open The Door'`
  - `slug = 'open-the-door'`
  - `description = 'Customers choose a door and reveal their coupon prize.'`
  - `icon = '🚪'`
  - `status = 'coming_soon'` or `active`
  - `default_spins`, `default_coupon_expiry_minutes`, flags, `sort_order`
  - `game_config = jsonb_build_object(...)`

### Automatic wiring in the target architecture

With the target architecture in place, the following happens automatically:

- `lib/games/index.ts` discovers `lib/games/open-the-door/contract.ts` and adds it to the registry.
- `GameType` includes `open_the_door` automatically.
- Builder selection UI includes the new contract card without manual code changes.
- The play page resolves `promotion.game_type === 'open_the_door'` to the new contract.
- The runtime page renders `OpenTheDoorRuntime` from the contract.
- Admin game management can display the new game based on the seeded `games` row and registry metadata.

## Proposed Target Architecture

### Proposed file structure

```
lib/games/
  open-the-door/
    contract.ts
    runtime.tsx
    builderPreview.tsx?      # optional
    configPanel.tsx?         # optional
  spin-wheel/
    contract.ts
    runtime.tsx
    builderPreview.tsx
    configPanel.tsx
  index.ts                  # auto-generated or convention-based registry
```

### Registry automation

`lib/games/index.ts` should:

- discover all game contract modules under `lib/games/*/contract.ts`
- export a single registry object keyed by `game.type`
- compute `availableGames`
- export helper functions like `getGameDefinition(gameType)`
- optionally export `GameType` as `keyof typeof gameRegistry`

### Database and admin

- `games` table should store `game_type` instead of depending on a human `slug`.
- `game_config` remains a generic JSONB config payload for runtime-specific defaults.
- The admin UI should join the persisted `games` row to the registry metadata by `game_type`.
- The game row should not duplicate core contract metadata unless the admin needs override fields like `display_name` or `description`.

### Runtime unification

- Merge the contract registry and the play runtime registry into one source of truth.
- `GameRuntimeRenderer` should resolve from the contract registry, not a separate `lib/game-pool/gameRegistry.ts` map.
- `resolvePromotionGame` should work off `promotion.game_type` and the same contract `GameType` union.

### Builder UI

- Use `availableGames` from the registry for game selection cards.
- Avoid hard-coded `BuilderGameType` unions and case statements.
- If a new game needs a special builder preview, it can provide `components.BuilderPreview` in the contract.

### Seed automation

- Add a seed generator script or migration template that reads contract metadata and populates the `games` table.
- Example automation: a Node script scans `lib/games/*/contract.ts` and writes a migration or JSON seed file with `game_type`, `name`, `icon`, `description`, `status`, and defaults.

## Conclusion

The ideal future state is: adding a new game requires only:

1. `lib/games/<new-game>/contract.ts`
2. `lib/games/<new-game>/runtime.tsx`
3. a seeded `games` database row with `game_type`

All registration, selection UI, runtime resolution, and admin integration should be automatic because the registry is convention-driven and single-source-of-truth.

This approach minimizes manual wiring, removes duplicate type definitions, and makes the platform scalable for many future game additions.
